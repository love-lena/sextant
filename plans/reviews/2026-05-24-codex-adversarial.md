**Top 5 Before Phase 1**
1. **BLOCKER:** Auth is contradictory: NATS requires JWT for every connection, while operator/client specs say no token, Unix socket perms only.
2. **BLOCKER:** `plans/bootstrap.md` orders NATS/JWT/config work before the CA/config owner exists.
3. **HIGH:** ClickHouse dedup is specified incorrectly; the table schema will not dedupe by event UUID.
4. **HIGH:** The client subscription API cannot resume from JetStream sequence because it returns only `Envelope`, with no stream metadata.
5. **HIGH:** Switchover happens before self-update and test environments, despite the docs claiming those are part of post-switchover viability.

**8. Bonus: Design Questions Worth Challenging**
- **Riskiest assumption:** “Everything on NATS + ClickHouse” will stay simple. The current docs push RPC, events, telemetry, debug firehose, user input, KV state, locks, and MCP-ish tool calls through one substrate without fully specifying acking, dedup, auth, or metadata. I would bet against this remaining boring unless the protocol contracts are tightened before M2-M7.

- **Second bet against:** “Full firehose, filter later.” It sounds operationally elegant, but it combines unbounded agent output, secrets, long retention, and weak redaction. That is the kind of design that works in demos and becomes a liability once real agents touch real repos and credentials.

- **Third bet against:** switchover before self-update/test envs. Agents can land code, but without isolated E2E and a deploy/rollback path, the first self-hosted development loop will depend on manual operator discipline, which contradicts the autonomy story.
