# ClickHouse — component spec

## Role

Long-term queryable store. ClickHouse holds:
- All bus events (indefinite retention, per-table policies)
- OTel-shaped telemetry (traces, metrics, logs)
- Audit log (long-term, queryable for forensics)
- Agent definition version history

See `architecture.md` §8 and §13 for design context.

## Version

ClickHouse 24.x+ (required for recent OTel schema improvements). Install via `brew install clickhouse` or pull binary.

## Config

Generated by `pkg/clickhouseboot/` from sextant config. Key settings:

- Data dir: `~/.local/share/sextant/clickhouse/`
- Listen: TCP localhost (HTTP + native protocol), no external interface
- No replication for initial single-host deployment
- Memory: configurable, default modest (sextant scale is small)

## Tables

Schema migrations live in `pkg/clickhouseboot/migrations/NNN-name.sql`, applied at sextantd startup in numeric order.

Initial tables (M3 milestone):

### `events` — generic bus events

```sql
CREATE TABLE events (
    id UUID,
    ts DateTime64(6),
    subject String,
    from_kind String,
    from_id String,
    to_kind String,
    to_id String,
    trace_id UUID,
    span_id UUID,
    kind String,
    payload JSON,
    INDEX idx_subject subject TYPE bloom_filter GRANULARITY 4
) ENGINE = ReplacingMergeTree(ts)
ORDER BY (id);
```

`ReplacingMergeTree(ts)` keeps the row with the highest `ts` per `ORDER BY` key on background merge. Re-inserts of the same `id` from shipper redelivery collapse over time. Queries that require strict dedup must use `FINAL` (slow) or `argMax(..., ts)` aggregation. For initial scale we accept the eventual-consistency tradeoff. Subject filtering is handled by the bloom filter index, not ORDER BY.

### `audit` — auth-relevant actions

```sql
CREATE TABLE audit (
    id UUID,
    ts DateTime64(6),
    actor String,
    agent_uuid UUID,
    action String,
    capability_required String,
    result String,
    payload JSON
) ENGINE = MergeTree()
ORDER BY (ts, actor, agent_uuid);
```

### `telemetry_traces`, `telemetry_metrics`, `telemetry_logs` — OTel-shaped

Adopt the [OpenTelemetry ClickHouse exporter schema](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/exporter/clickhouseexporter) for the three telemetry tables — it is well-tested, documented, and lets us drop OTLP data in via that exporter later without schema migration.

`telemetry_traces` (matches OTel span data model):

```sql
CREATE TABLE telemetry_traces (
    Timestamp DateTime64(9),
    TraceId String,
    SpanId String,
    ParentSpanId String,
    TraceState String,
    SpanName LowCardinality(String),
    SpanKind LowCardinality(String),
    ServiceName LowCardinality(String),
    ResourceAttributes Map(LowCardinality(String), String),
    SpanAttributes Map(LowCardinality(String), String),
    Duration Int64,
    StatusCode LowCardinality(String),
    StatusMessage String,
    Events Nested(Timestamp DateTime64(9), Name LowCardinality(String), Attributes Map(LowCardinality(String), String)),
    Links Nested(TraceId String, SpanId String, TraceState String, Attributes Map(LowCardinality(String), String))
) ENGINE = MergeTree()
ORDER BY (ServiceName, SpanName, toUnixTimestamp(Timestamp));
```

`telemetry_metrics` and `telemetry_logs`: follow the same exporter schemas. Migration files in `pkg/clickhouseboot/migrations/` carry the full DDL.

### `agent_definitions_history` — every mutation

```sql
CREATE TABLE agent_definitions_history (
    agent_uuid UUID,
    version UInt64,
    ts DateTime64(6),
    actor String,
    change_kind String, -- create | update | archive | restore
    definition JSON
) ENGINE = MergeTree()
ORDER BY (agent_uuid, version);
```

## Migrations

Numbered SQL files, idempotent re-runs. Tooling: `pkg/clickhouseboot/migrate.go` reads the migrations dir at startup, checks a `sextant_migrations` table, applies pending migrations in order.

## Retention

Per-table TTL clauses, tuned per data class:

- `events`: TBD (default forever, can be reduced if disk pressure)
- `audit`: forever
- `telemetry_traces`: 30 days
- `telemetry_metrics`: 90 days
- `telemetry_logs`: 30 days
- `agent_definitions_history`: forever
