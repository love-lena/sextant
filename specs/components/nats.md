# NATS — component spec

## Role

Universal event bus for sextant. Provides:
- **JetStream streams**: persistent event log with consumer groups + replay
- **JetStream KV**: current-state storage (agent definitions, viz specs, `ui.state.*`, worktree registry, deploy locks, test envs)
- **Request/reply**: substrate for RPCs

See `architecture.md` §6 for the control plane design.

## Version

NATS server 2.10+ (required for JetStream KV and recent reliability features). Install via `brew install nats-server` or pull binary.

## Config

Generated by `pkg/natsboot/` from sextant config. Key settings:

- JetStream enabled with file-based storage at `~/.local/share/sextant/nats/`
- One TCP listener, two credential paths. NATS Server does not have a native Unix-socket transport, so the architecture's "Unix-socket-as-trust-boundary" is realized with **TCP on `127.0.0.1` + Unix-perm-protected credentials** rather than a literal socket. Loopback-only binding plus host-firewall norms keep the listener off the network.
  - **Operator path**: NATS user `operator` with a random password generated at `sextant init` and written to `~/.config/sextant/operator.creds` (mode `0600`, owned by the operator's Unix user). The CLI/TUI/sextantd load this creds file when connecting. The Unix file permissions on the creds file are the trust boundary. The `operator` user has full publish/subscribe authority on every subject and full JetStream admin.
  - **Agent path**: per-incarnation NATS user identities authenticated via JWT signed by the sextant CA (M5+). The TCP listener's `nkeys`/`jwt` config trusts the CA's public key; sidecars connect with their issued JWT. Subject ACLs are encoded in the JWT.
- Anonymous access is forbidden. Every connection must authenticate as either the `operator` user (creds file) or via a CA-issued JWT (M5+).
- M2 ships only the operator path. JWT verification config references the CA public-key file; the file is allowed to be empty/missing in M2 and gets populated by `sextant init` in M5. Agents do not connect to NATS until M11.
- NATS bind: `127.0.0.1:<port>` (default 4222; reconfigurable). The listener is never bound to a routable interface for initial. Containers reach the listener via `host.docker.internal` resolving to the loopback host (OrbStack / Docker Desktop default).
- Sextantd's own control socket (`sextantd.sock`) remains a true Unix socket; that is a separate sextantd-side surface and is unaffected by NATS' lack of socket support.

| `user_input` | `user_input.>` | 30 days | TBD |
| `control_rpc` | `sextant.rpc.>` | 1 day | TBD |

Wildcard rules: `*` matches exactly one token; `>` matches one or more. Multi-token subjects (e.g. `telemetry.metrics.shipper.lag_seconds`) require `>` on the listening pattern, else messages are silently dropped from the stream.

Retention numbers are starting points; tune based on disk usage.

## KV stores to create

| Bucket | Purpose | Notes |
|---|---|---|
| `agent_definitions` | current agent definition per UUID | watchable |
| `templates` | agent templates | shipped defaults + operator-added |
| `viz_specs` | visualization specs | post-M17 feature |
| `ui_state` | inter-UI coordination | per-operator-scoped keys |
| `worktrees` | worktree registry | metadata per worktree |
| `locks` | mutual-exclusion locks | keys: `merge` (TTL 5 min), `deploy` (TTL 10 min); add more as needed |
| `test_envs` | test environment registry | TTL per env |

**Lock conventions**: all mutex-style locks live in the single `locks` bucket. Lock keys are short verbs (`merge`, `deploy`, ...). Holders write their UUID/host as the value; expired TTL = lock auto-released. References elsewhere (`architecture.md` §11, §12; `conventions/git-workflow.md`) use the form `locks.<key>` (e.g. `locks.merge`).
